# 第3章 系统需求分析

## 3.1 用户与智能生活管家业务需求分析

### 3.1.1 应用场景概述

信息碎片化是当代人日常生活的普遍困扰。工作任务散落在邮件和即时通讯工具中，日程安排记录在多个日历应用里，出行信息分散在地图和旅游平台。这种分散存储的模式导致用户需要频繁切换应用以获取完整信息，手动整理和归纳这些数据耗费大量时间，信息量增长后仅凭记忆难以有效检索。

LifePilot 定位为智能生活管家，通过自然语言这种最符合人类思维习惯的方式，帮助用户统一管理日常事务。用户通过对话即可完成任务创建、查询和修改，同时系统基于私有知识库提供精准的问答服务。

系统的核心交互界面是 LifePilot Web 应用。用户通过这个界面与 AI 管家对话、管理任务和文档、查看日历。Metaphorical 和 OmniBase 是开发过程中的配套工具，前者用于日志监控便于问题排查，后者用于数据库管理辅助开发调试。这两个配套项目不属于用户需求范畴，后文不再赘述。

### 3.1.2 用户特征分析

LifePilot 的目标用户主要是需要管理日常任务和知识的个人用户。

日常任务管理是大多数用户的核心需求。用户需要记录待办事项、设置时间提醒、追踪完成状态。这类用户期望操作简便、响应及时，能够用最少的步骤完成任务管理。

知识工作者群体有文档管理和检索需求。他们日常阅读大量资料，希望建立一个私有知识库，通过对话快速定位所需信息。这类用户关注检索准确性和知识安全性。

移动场景用户倾向于语音交互。在双手不便操作键盘的场景下，语音输入能大幅提升效率。这类用户关注 ASR 识别率和 TTS 播报质量。

### 3.1.3 业务流程分析

LifePilot 的核心业务流程以自然语言对话为入口。

当用户描述一个任务需求时，系统通过意图识别判断用户想要的操作类型。如果识别为任务创建，系统调用任务管理服务解析关键信息，生成结构化任务计划呈现给用户确认。引入人在回路机制，AI 生成的计划不会直接执行，而是等待用户最终确认。用户接受则写入数据库并设置提醒，拒绝则 AI 重新生成。

知识问答链路中，用户先上传文档，系统自动完成解析、分块、向量化存储。用户提问时，系统从向量数据库、关键词索引和知识图谱三个渠道并行检索相关片段，重排后送入大语言模型生成答案。

出行规划场景下，系统进入 ReAct 推理模式，由大语言模型自主决定调用高德地图和小红书 MCP 工具的时机和顺序，获取地点信息和旅游攻略后整合返回。

### 3.1.4 系统边界

LifePilot 需要与以下外部服务协作：

LLM 服务商（OpenAI、DeepSeek）提供对话生成和意图识别能力。SiliconFlow 平台提供文本向量化和重排能力。高德地图和小红书为出行规划提供数据。七牛云 OSS 存储用户上传的文档。邮件服务商发送任务提醒。

数据存储方面，MySQL 存储用户账户和任务数据，Redis 承担会话缓存和任务队列，MongoDB 存储对话历史，Qdrant 和 Neo4j 分别作为向量检索和知识图谱的存储引擎。

### 3.1.5 需求目标转化

从用户痛点可以提炼出系统的核心需求目标。

多应用切换的繁琐，需要统一对话入口来解决，所有操作通过自然语言完成，无需在多个应用间切换。

手动整理信息的时间成本，需要 AI 自动处理来降低，系统从自然语言描述中提取关键信息，自动生成结构化任务安排。

知识检索的困难，需要 RAG 技术来解决，基于私有文档的精准问答提升信息获取效率。

任务遗漏的风险，需要多维度提醒机制来规避，应用内通知和邮件通知的组合覆盖多种场景。

语音交互体验的不足，需要流式 ASR/TTS 技术来改善，边说边转、边生成边播报的体验接近自然对话。

## 3.2 核心功能需求分析

### 3.2.1 任务管理

任务管理是 LifePilot 最基础的功能模块。

任务创建需要支持从自然语言描述中提取任务要素。系统识别标题、描述、开始时间、截止时间、优先级、标签等字段。还需要支持重复任务，按日、周、月周期自动生成后续任务实例。

任务查询支持多维度筛选。可以按状态查看待办事项，按时间范围查看某日或某周任务，按标签查看特定类别的所有任务。支持自然语言查询，如直接问"今天有什么任务"。

任务变更支持修改任意字段。时间调整是最常见的需求，用户可能需要推迟截止时间。状态变更是高频操作，完成任务后标记为已完成。标签的增删改查同样需要支持。

任务提醒提供灵活配置。用户可设置提醒提前量（如提前 30 分钟、提前 1 天）和重复提醒次数。提醒触发后通过应用内通知和邮件两种方式投递。

### 3.2.2 AI 对话

对话是用户与系统交互的主要界面。

对话生成通过 LangGraph 工作流引擎处理用户输入。不同请求类型进入不同工作流分支：任务相关进入任务处理分支，知识问答进入 RAG 分支，出行规划进入 ReAct 分支。回复采用 SSE 协议流式传输，用户无需等待完整生成就能看到中间结果。

意图识别是分流的前提。用户输入后调用 DeepSeek 模型进行分类，输出意图标签（build_todo、update_todo、rag、travel 等）决定后续工作流走向。

任务规划场景采用三角色协作模式。执行者负责理解需求并生成结构化任务计划。检查者负责验证输出的完整性、合理性和格式规范性，不通过则返回执行者重试。管家负责将验证通过的计划以友好方式呈现给用户，等待确认。

人在回路机制确保 AI 生成结果的安全性。系统不会直接执行 AI 生成的计划，而是暂停工作流等待用户确认。用户接受则写入数据库，拒绝则返回执行者重新生成。

### 3.2.3 RAG 知识库

知识库功能支持私有文档管理和智能问答。

文档上传支持多种格式。PDF 通过 PyMuPDF 和 pdfplumber 结合提取文本。Word 使用 python-docx 解析，Excel 使用 openpyxl，PPT 使用 python-pptx。图片通过 OCR 识别文字。纯文本和 Markdown 直接读取。

分块处理将长文档切分为适合检索的短片段。块太小可能导致语义不完整，块太大可能降低检索精度。系统采用固定窗口加分隔符的方式，并记录块之间的关联关系。

文档向量化使用 SiliconFlow Qwen3-Embedding-8B 模型将文本块转为高维向量存入 Qdrant。同时调用大语言模型抽取实体和关系三元组存入 Neo4j 图数据库。

智能问答流程中，系统并行执行向量检索、关键词检索和图检索三种操作。向量检索用问题向量在 Qdrant 中寻找语义相似的文档块。关键词检索用 BM25 算法匹配问题关键词。图检索从 Neo4j 中找到与问题实体相关的三元组。三路结果经 Cross-Encoder 重排后取最相关的片段作为上下文，由 MiniMax-M2.7 生成答案。答案标注信息来源，用户可追溯原文。

项目级数据隔离确保不同用户的文档互不干扰。每个应用使用独立的 Qdrant collection 和 Neo4j 数据空间，通过 app_key 标识和隔离。

### 3.2.4 语音交互

语音交互包括语音识别（ASR）和语音合成（TTS）两个方面。

ASR 流程中，用户点击录音按钮后，浏览器调用 MediaRecorder API 采集音频流并编码为 webm 格式。采集完成后 POST 到 ai-server 转写接口，SiliconFlow SenseVoiceSmall 模型进行中文普通话识别，结果返回后自动填入对话框并触发对话流程。

TTS 采用后端代理模式。用户开启语音播报后，对话回复在生成同时送往 ai-server TTS 接口。服务器返回流式音频数据，前端通过 Audio 标签边收边播。后端代理模式的考虑是保护 API Key 不暴露给浏览器，同时可预处理过滤不适合播报的格式符号。

### 3.2.5 出行规划

出行规划功能依赖 ReAct 推理模式和大语言模型的工具调用能力。

当用户提出出行需求时，LangGraph 工作流加载高德地图和小红书 MCP 工具。模型在推理过程中自主决定调用时机和顺序。如果只询问地点基本信息，可能只调用一次地图搜索。如果要比较不同路线，会多次调用路线规划 API 获取对比数据。如果还想了解旅游体验，会额外调用小红书攻略搜索。

最终答案由模型用自然语言整合各类信息返回，而不是简单罗列 API 结果。

### 3.2.6 自我记录

自我记录功能为用户提供视频形式的日记记录能力，支持在录制过程中叠加个性化贴纸装饰。

功能包含贴纸库管理和视频录制两大模块。贴纸库支持用户上传 PNG 格式图片作为贴纸素材，系统对贴纸进行统一管理，包括增删改查操作。每张贴纸可设置颜色属性，用户在录制时能够从贴纸库选择素材添加到画面中。

视频录制采用画布合成方案。摄像头采集的视频流与用户添加的贴纸在 Canvas 画布上实时合成，贴纸位置和大小通过拖拽调整，分辨率采用 720p（1280×720）标准。录制完成后使用 MediaRecorder API 将画布内容编码为 webm 格式，音频轨道同步混合进最终文件。录制生成的视频通过七牛云 OSS 存储，前端直传模式减少服务器带宽压力。

视频播放管理提供录制历史查看和在线播放功能。用户可查看近期录制的视频列表，支持在线预览播放。视频元数据（含 OSS 地址、时长、创建时间）通过后端 API 持久化存储。

### 3.2.7 功能优先级

P0 级别是核心功能：自然语言创建任务、任务列表查看、任务状态变更、人在回路确认、流式对话响应、意图识别路由、文档上传解析、RAG 智能问答。

P1 级别是重要增强：任务提醒设置、三角色协作生成、文档删除管理、ASR 语音转文字、TTS 流式播报。

P2 级别是增值功能：地点搜索、路线规划、旅游攻略搜索、自我记录视频日记。

## 3.3 性能与可靠性需求分析

### 3.3.1 性能需求

不同类型操作有不同的性能考量。

同步 API 操作（用户注册、任务 CRUD 等）响应时间应控制在 500 毫秒以内。瓶颈主要在数据库 IO 和网络延迟。

页面加载首屏时间应控制在 2 秒以内。Next.js 的服务端渲染能力可以有效缩短这段时间。

AI 对话的首字响应时间应控制在 3 秒以内。这是用户感知即时的临界值。

RAG 问答端到端时间目标为 10 秒以内。检索和重排约 300 毫秒，生成时间取决于模型。

文档上传（10MB 以内）处理时间目标为 10 秒以内。OCR 和格式解析是主要耗时点。

系统并发设计目标为 100 用户同时在线，RAG 接口吞吐量 10 QPS，文档上传接口 5 QPS。

### 3.3.2 可用性需求

核心功能（AI 对话、任务管理）可用性目标为 99.5%，每月允许停机约 3.6 小时。RAG 服务可用性目标为 99%，每月允许停机约 7.3 小时。

容错设计方面，LLM 服务不可用时对话功能降级为简单回复。Qdrant 不可用时 RAG 降级为纯向量检索模式。Redis 不可用时任务调度降级为同步检查。

数据备份方面，MySQL 每日凌晨全量备份保留 30 天。MongoDB 每日增量加每周全量备份保留 30 天。Qdrant 和 Neo4j 每周全量备份。

### 3.3.3 可靠性需求

任务数据保证强一致性，写入成功后才返回响应。RAG 数据采用最终一致性，文档入库后 5 分钟内可检索。消息通知至少一次投递。

安全性方面，用户密码 bcrypt 加密存储。敏感配置通过环境变量注入，不写入代码仓库。通信使用 HTTPS。API Key 仅在后端使用，不暴露给前端。

日志追踪通过 yanmengs-logs SDK 实现，异步上报不阻塞主业务。SDK 做好错误处理，上报失败不影响主应用。

### 3.3.4 可维护性需求

开发环境通过 pnpm dev 启动各模块，便于调试热更新。生产环境 Docker 容器化部署，Nginx 反向代理将域名请求转发到对应容器端口。

Metaphorical 日志平台用于问题排查，支持按项目和时间筛选日志，便于定位异常。
