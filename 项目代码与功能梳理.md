## 2. 项目定位

从代码结构和功能实现来看，LifePilot 不是单一的待办清单系统，而是一个围绕“任务管理 + 智能对话 + 私有知识库 + 多模态交互”构建的复合型个人助理平台。系统的核心目标有三类：

- 将传统待办事项管理扩展为带提醒、统计、标签、优先级和时间管理的个人任务中心
- 将大模型能力嵌入任务规划、知识问答、出行建议、旅行攻略等场景
- 将文本、语音、文件、知识库、邮件提醒、数字人等能力接入到同一套产品体系中

从使用方式看，用户既可以手动创建和管理任务，也可以通过聊天输入、语音输入、知识库问答来获得智能辅助。系统不仅支持 CRUD，更强调“自然语言驱动的任务生成与执行建议”。

## 4. 系统整体架构

### 4.1 主业务架构

```text
用户浏览器
  └─ LifePilot 前端
      ├─ Next.js 页面与组件
      ├─ MobX 状态管理
      ├─ 内置 API Route
      ├─ 七牛文件上传
      └─ AI 抽屉 / 语音输入 / 知识库页面

LifePilot 前端
  ├─ 调用本地 API Route
  │   ├─ MySQL：任务、标签、用户、文件、录屏、贴图
  │   ├─ MongoDB：错误日志
  │   └─ Redis：认证码、登录态、任务调度
  ├─ 调用 LifePilotServer
  │   └─ 智能对话、SSE、TTS 代理、工作流恢复
  └─ 调用 ai-server
      ├─ 文档解析与知识入库
      ├─ RAG 问答
      ├─ ASR 语音转写
      └─ OSS Token

LifePilotServer
  ├─ LangGraph 路由与任务工作流
  ├─ MCP 客户端调用 LifePilot_mcp
  ├─ Amap MCP / 小红书 MCP
  ├─ MongoDB：聊天消息明细
  ├─ MySQL：聊天会话标题
  └─ Redis：任务提醒调度

LifePilot_mcp
  └─ 以 MCP 工具形式暴露任务与标签操作

ai-server
  ├─ 文档解析器
  ├─ Embedding 服务
  ├─ Qdrant 向量检索
  ├─ Neo4j 图谱抽取
  ├─ Weaviate 文档管理兼容层
  ├─ TTS / ASR
  └─ App Key 生成与校验
```

### 4.2 关键交互链路

#### 链路一：用户手动创建任务

`LifePilot 页面 -> /api/task -> Prisma MySQL -> Redis 调度集合`

如果任务开启提醒并设置截止时间，系统会把提醒触发时间写入 Redis 的有序集合 `scheduler:tasks`，后端调度器会轮询该集合并发送提醒邮件。

#### 链路二：用户通过 AI 生成任务

`AI 抽屉 -> LifePilotServer SSE -> LangGraph 路由 -> build_todo 工作流 -> 用户确认 -> MCP create_task -> MySQL/Redis`

这是系统里最有论文价值的一条链路，因为它体现了自然语言到结构化任务的转换、审查、确认和落库过程。

#### 链路三：知识库文件上传

`前端选择文件 -> 七牛上传 -> /api/knowledge 记录文件元数据 -> ai-server /documents/process-url -> 解析/切块/向量化/图谱抽取 -> 回调 /api/file 更新状态`

前端页面上会展示 `ragIng / success / fail` 三种入库状态，失败时会记录错误信息并支持重试。

#### 链路四：语音对话

`前端录音 -> 七牛上传音频 -> ai-server ASR 转文本 -> LifePilotServer 对话 -> 返回文本流 -> ai-server TTS 合成播放`

#### 链路五：任务提醒

`任务写入 Redis 调度集合 -> LifePilotServer scheduler 轮询 -> 判断是否提醒或超时 -> QQ 邮箱发送 HTML 邮件`

## 6.5 AI 助手对话

### 功能范围

- 对话式任务建议
- 对话式任务生成
- 知识库问答
- 出行规划
- 旅行攻略建议
- 语音对话输入
- 聊天历史查看
- AI 输出 TTS 播放
- 任务生成结果的人工确认

### 前端实现

AI 抽屉主组件位于：

- `LifePilot/src/app/components/drawer.tsx`

它是整个智能交互的前端核心，负责：

- 输入框与消息流展示
- SSE 消息解析
- 展示模型的 `think` 与 `response`
- 历史会话读取
- 语音录音上传
- TTS 播放
- 对 `confirm` 状态展示“接受/拒绝”按钮

### 后端实现

`LifePilotServer/routes/chat.ts` 提供三类能力：

- `POST /chat/chat`：主对话入口，SSE 流式返回
- `GET /chat/history/list`：会话列表
- `GET /chat/history/message`：消息明细
- `POST /chat/chat/judgment`：任务生成后的 accept/reject 回传

### 对话数据存储方式

这里采用了双存储设计：

- MySQL `ChatHistory` 表：保存会话标题、会话 ID、用户 ID
- MongoDB `ChatHistory` 集合：保存逐条消息内容、角色、思考过程、音频链接、时长

这一设计的意图很明确：

- 会话列表适合结构化查询，用 MySQL
- 对话消息内容更灵活，字段可扩展，用 MongoDB

### 流式返回机制

系统没有使用 EventSource，而是使用 `fetch + ReadableStream` 手动解析 SSE。这样做的好处是：

- 前端可以统一控制流式事件类型
- 可以接收自定义事件，如 `id`、`think`、`response`、`confirm`、`thread_id`、`refresh`
- 便于在同一条对话链路里插入人工确认节点

## 6.6 智能工作流编排

`LifePilotServer/langgraph` 是智能能力的核心实现区。

### 路由器工作流

入口在 `workflow/router.ts`，会先调用模型判断用户意图，然后路由到五种工作流之一：

- `build_todo`：从自然语言中生成待办任务
- `what_to_do`：根据现有任务与上下文给出行动建议
- `trivel`：旅行攻略或出行计划建议
- `howToGo`：路线规划
- `rag`：知识库问答或普通问答兜底

### `build_todo` 工作流

这是最重要的一条链路，流程为：

1. 读取当前时间
2. 加载历史对话与现有任务
3. `executor` 生成结构化任务 JSON
4. `inspector` 审查结果是否合格
5. `formatter` 整理为用户可读文本
6. `user_judgment` 触发前端确认
7. 用户接受后调用 MCP `create_task`
8. 刷新任务列表

其中特别值得写入论文的点有两个：

- 使用了“执行者-审查者”双节点结构，而不是单次直接生成
- 使用 `interrupt` 和 `resume` 实现人工确认，避免模型直接写入任务

### `what_to_do` 工作流

基于现有任务、当前时间和历史上下文，为用户生成行动建议。结构同样包含：

- executor
- inspector
- manager

其中 `manager` 负责输出更适合展示的最终答复。

### `trivel` 与 `howToGo`

这两条工作流会启用工具调用模式：

- 本地 MCP 工具
- 高德地图 MCP
- 小红书 MCP

因此它们适合处理路线规划、地点查询、攻略整理等需要外部工具信息的场景。

### `rag`

先向 `ai-server` 发起知识库问答；若未命中，则退化为普通大模型问答。

## 6.7 MCP 工具层

### 作用

`LifePilot_mcp` 的作用不是直接服务最终用户，而是给智能工作流提供“可执行工具”。

### 暴露的工具

当前注册的 MCP 工具包括：

- `create_task`
- `get_tasks`
- `update_task`
- `delete_task`
- `create_tag`

### 作用方式

当 LangGraph 工作流生成结构化任务后，不会直接操作数据库，而是通过 `LifePilotServer/utils/mcp.ts` 中的 MCP 客户端去调用 `LifePilot_mcp`。这层设计把“智能体决策”与“业务数据写入”分离开了。

### 价值

这一层非常适合写成论文中的“工具调用层”或“执行层”：

- 上层只负责推理和决策
- 下层只负责受控的数据操作
- 接口边界清楚
- 后续可扩展更多工具

## 6.8 知识库管理

### 功能范围

- 文件上传
- 文件列表查看
- 文件重命名
- 文件删除
- 文件解析状态展示
- 文件解析失败重试
- 基于私有文档的问答

### 前端实现

页面位于：

- `LifePilot/src/app/home/knowledge/page.tsx`

关键组件：

- `FileGrid.tsx`
- `GridUploadCard.tsx`

### 上传流程

1. 前端通过 `qiniuStore` 并发上传文件到七牛
2. 上传成功后调用 `/api/knowledge` 记录文件信息
3. 记录表为 MySQL `folder`
4. 再调用 `yanmengsClient.knowledge.processUrl`
5. `ai-server` 下载文件、解析文件、入库知识库
6. 入库状态通过回调 `/api/file` 写回 `alreadyRag` 与 `ragError`

### 文件状态

知识库文件当前使用以下状态字段：

- `ragIng`
- `success`
- `fail`

页面上会显示对应状态标签，并在失败时支持再次触发解析。

#### 代码中的新实现

- Neo4j 图谱存储

#### 代码中的兼容层

`main.py`、`api/document.py`、`core/db.py` 仍然保留了 Weaviate 客户端初始化与部分文档管理逻辑。尤其是：

- 文档删除时会先检查 Weaviate collection
- `main.py` 的标题仍写着 `Weaviate RAG API`

#### 因此可以得出的结论

当前 `ai-server` 更像一个处于演进中的混合版本：

- 问答与索引主流程已经朝 `Qdrant + Neo4j + SiliconFlow Embedding` 迁移
- 文档管理接口仍保留部分 Weaviate 兼容逻辑

后续论文写作必须统一口径，不能一处写 Weaviate，一处写 Qdrant/Neo4j。建议按“当前主检索链路”来写：

- 向量层：Qdrant
- 图谱层：Neo4j
- Embedding：SiliconFlow
- 文档管理兼容层：保留 Weaviate 接口遗留逻辑

### RAG 查询流程

`rag/workflow/rag_pipeline.py` 的核心流程为：

1. Query Analyzer 判断意图与优化查询
2. Dense Retriever 执行向量检索
3. Sparse Retriever 执行关键词检索
4. Graph Retriever 执行图谱检索
5. FilterMergeNode 融合与过滤结果
6. RerankerNode 执行交叉编码器重排
7. GeneratorNode 基于检索结果生成回答

### 文档入库流程

1. 解析 PDF、Word、TXT、Excel、PPT、图片
2. 分块
3. 生成 Embedding
4. 写入向量库
5. 抽取实体关系三元组
6. 写入图数据库

### 应用隔离机制

`app-key` 通过 `application.py` 进行生成与解密，最终映射到合法的 `app_name / collection_name`。这样不同应用的知识库可以物理隔离。

## 7. 关键实现机制

## 7.1 前端架构

前端采用 Next.js App Router，页面和本地 API Route 在同一工程中维护。其主要特点有：

- 页面层负责交互与可视化
- `MobX Store` 管全局状态
- `lib/apis` 统一封装请求
- `app/api` 提供本地业务接口
- `prisma`、`mongoose`、`redis` 客户端在服务端侧统一接入

这种结构兼顾了开发效率与模块清晰度，对毕业设计项目来说是比较实用的方案。

## 7.2 聊天流式传输

系统的流式对话不是简单地返回整段文本，而是拆分成多类事件：

- `id`
- `think`
- `response`
- `confirm`
- `thread_id`
- `refresh`
- `done`

这让前端可以分别展示：

- 思考过程
- 最终答案
- 任务确认按钮
- 任务列表刷新

这也是系统能支持“人工确认写任务”的基础。

## 7.3 提醒调度

提醒调度使用 Redis 有序集合实现。其核心思路是：

- 任务创建或修改时，根据截止时间计算提醒时间
- 把任务 ID 按时间戳写入 `scheduler:tasks`
- 后端常驻轮询，拿出当前时刻之前的任务
- 若任务已超时，更新状态并发送超时邮件
- 若尚未超时，则发送提醒并计算下一次提醒时间

这一机制比简单的 `setTimeout` 或单机定时任务更适合服务化部署。

## 7.4 知识库状态回写

知识库导入不是同步阻塞式操作，而是“先记元数据，再异步解析，再回写状态”：

1. 前端先能看到文件
2. 状态先设为 `ragIng`
3. 后台解析完成后回调更新
4. 失败时保留错误信息

这让用户体验更稳定，也让页面具备重试能力。

## 7.5 双消息存储

聊天模块把“会话元数据”和“消息内容”拆开存储：

- MySQL 适合会话列表与关联查询
- MongoDB 适合消息内容与扩展字段

这一点后续写论文时可以概括为“按数据结构特征选择存储介质”。
