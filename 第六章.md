# 第6章 系统架构与部署

系统进入上线阶段后，设计重点从功能实现转向运行稳定性、发布可控性与维护成本。前文已完成前端、业务服务、工具服务与 AI 服务的功能划分，本章将部署目标限定为单台 Ubuntu 服务器可持续运行，并支持后续版本迭代。部署链路采用四层组织方式：宿主环境提供统一运行基础，Docker 承载服务实例，Jenkins 负责构建发布流程，Nginx Proxy Manager 负责公网入口、域名转发与 HTTPS 接入。该组织方式将系统变更收敛到镜像与流水线层，便于回溯与回退。

## 6.1 部署架构设计

部署架构围绕“可复现、可回退、可观测”三点展开。可复现要求同一编排文件能够在新节点快速恢复服务；可回退要求发布失败时可在短时间切回上一版本；可观测要求每次发布都有完整日志与状态记录。图 6-1 与图 6-2 对应部署拓扑和发布流水线。

### 6.1.1 宿主环境与 Docker 容器组织

部署节点采用 Ubuntu 长期维护版本。该平台在容器生态、自动化工具与运维脚本方面具备完整支持，版本维护周期也与项目运行周期匹配。部署初期先完成宿主机层面的统一配置，包括软件仓库更新、时区校准、系统账户分级、远程访问策略与基础网络策略。此阶段不引入业务服务，目的是将系统层问题与应用层问题拆分处理。

宿主机资源按最小可运行原则分配。系统层只保留容器引擎、持续集成执行端与反向代理管理工具所需组件，业务运行时依赖统一放入镜像。该做法降低了环境漂移概率，即便后续迁移到新节点，也可通过同一镜像与编排文件恢复运行状态。

系统服务采用 Docker 容器化封装。前端服务、业务服务、工具服务与 AI 服务分为独立容器实例，通过容器网络完成内部通信，对外仅暴露代理层需要的入口端口。该结构避免了不同服务在同一运行空间内的依赖冲突，服务升级时不需要在宿主机逐项替换依赖。

容器编排使用声明式配置管理，配置项覆盖镜像版本、启动顺序、环境参数、重启策略与持久化目录。服务上线过程遵循固定顺序：先加载配置，再启动依赖服务，最后开放对外访问。发布新版本时以镜像替换为主路径，不在运行容器内部执行临时修改。

### 6.1.2 Jenkins 持续集成与发布流程

发布流程由 Jenkins 统一调度。流水线阶段由代码拉取、镜像构建、镜像推送、目标节点部署、健康检查组成。每次执行都生成独立构建记录，记录内包含提交版本、执行编号与阶段状态。该记录机制将发布行为转化为可追踪对象，避免仅凭手工命令历史回溯线上变更。

自动化发布的关键作用在于减少重复性人工操作。若采用手工发布，常见风险包括命令遗漏、版本误用与多服务更新顺序错位。流水线将这些步骤固化后，发布稳定性由流程一致性保障。部署阶段采用先启动新实例、后切换入口流量的顺序，健康检查失败时终止切换并保留旧实例。

系统部署过程中除容器服务外，还复用两个已发布的 npm 包：`yanmengs-logs` 与 `yanmengs-rag-package`。前者用于统一日志上报，后者提供 AI 相关能力封装。两者属于代码依赖而非独立进程服务，不进入 Docker 编排，由应用在发布阶段安装并调用。

### 6.1.3 Nginx 统一入口与 HTTPS 接入

公网访问入口由 Nginx Proxy Manager 统一维护。每个业务入口配置为单独代理主机条目，条目内维护域名、目标地址、转发端口与访问策略。外部请求先到达 80 或 443 端口，再由代理层按域名路由到对应容器服务。该模式将入口管理与业务处理拆分，服务容器无需直接处理公网接入细节。

HTTPS 证书由 Let’s Encrypt 自动签发与续期。证书生命周期交由代理层统一管理后，业务服务只需关注接口逻辑与数据处理。域名或端口映射调整时，修改点集中在同一管理面板，变更路径短，验证步骤清晰。

当前生产环境采用四个公网域名，分别对应前端访问入口、业务服务入口、工具服务入口与 AI 服务入口。域名映射关系见表 6-1。

| 域名 | 目标地址 | 对应服务 | 访问属性 |
|---|---|---|---|
| lifepilot.website | 172.17.0.1:3000 | 前端服务 | Public，HTTPS |
| server.lifepilot.website | 172.17.0.1:5000 | 业务服务 | Public，HTTPS |
| mcp.lifepilot.website | 172.17.0.1:7000 | 工具服务 | Public，HTTPS |
| yanmengsss.xyz | 172.17.0.1:7100 | AI 服务 | Public，HTTPS |

表 6-1 生产环境域名与端口映射关系

四个入口分离后，访问路径与职责边界保持清晰。前端域名面向用户交互，业务域名处理主业务接口，工具域名承载工具调用能力，AI 域名承载模型与知识处理能力。服务边界清晰后，后续扩容可按域名入口对应的服务压力独立调整。

## 6.2 前端关键实现

前端实现关注两个问题：其一，长流程交互是否可见；其二，媒体处理链路是否可控。图 6-4 与图 6-7 分别对应事件协议状态流与视频上传管线。

### 6.2.1 记录画布渲染与状态同步

记录页面采用画布渲染方式组织用户交互状态。页面在空闲、录制、上传、完成四种状态之间切换，每次状态切换都伴随明确的按钮反馈与提示信息。该设计降低了长流程操作中的不确定感，用户可以在任意时点判断当前阶段是否正常推进。

前端与后端之间通过事件流进行状态同步。服务端将阶段性结果以事件类型返回，前端按事件类型更新界面区域与提示文本。图 6-4 所示的协议结构避免了等待完整响应后再统一刷新界面的滞后问题。

### 6.2.2 视频录制与上传管线

视频录制链路从设备授权开始，随后进入采集、封装、上传与回写阶段。前端在本地完成录制片段整理后提交对象存储，上传完成后再把资源地址与元数据回写业务服务。图 6-7 展示了该管线的关键节点。

该管线将“媒体上传成功”与“业务记录写入成功”拆成两个可观测阶段。若上传成功但回写失败，界面可给出补写提示；若上传失败，界面可立即终止流程并引导重试。

## 6.3 后端关键实现

后端实现主要承担检索服务与调度服务两条链路。前者负责知识入库与检索问答闭环，后者负责到期任务处理与中断恢复。图 6-3、图 6-5、图 6-6 对应这两条链路的关键状态。

### 6.3.1 RAG 检索服务

检索服务接收文档后，先执行解析与切分，再进入向量写入与索引构建。入库状态不直接绑定到单次请求生命周期，而是通过异步回写更新业务状态。图 6-6 展示了异步回写状态转换过程。

问答阶段以索引检索结果为证据输入，再生成回答文本。该路径把“证据召回”与“回答生成”分离处理，便于在回归测试中分别评估召回质量与回答质量。

### 6.3.2 提醒调度服务

调度服务采用队列驱动模型处理到期任务。任务进入队列后由调度器按时间窗口拉取，先执行抢占判断，再执行提醒动作或状态更新。图 6-5 所示流程用于控制并发条件下的重复处理风险。

当任务处理链路被中断时，服务根据已落库状态执行恢复逻辑，避免重复写入与状态回退。图 6-3 所示的中断恢复流程将恢复动作约束在可判定状态内，保证调度链路在异常后可继续推进。

## 6.4 本章小结

本章给出了系统上线阶段的部署组织与关键实现路径。部署层通过容器化、流水线化和统一入口管理控制发布风险；实现层通过前端状态同步与后端异步回写、调度恢复机制保证长流程可执行性。该结构为后续测试与迭代提供了稳定基础。
