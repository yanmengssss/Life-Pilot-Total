# 第2章 相关技术与开发工具

## 2.1 Web应用架构与前后端分离模式

### 2.1.1 前后端分离架构

传统 Web 应用采用服务端渲染模式，前端代码与后端逻辑高度耦合。前后端分离架构将前端和后端解耦为独立服务，各自使用擅长的技术栈，通过 HTTP 协议进行数据交互。

项目采用典型的前后端分离架构。前端使用 Next.js 构建单页应用，后端使用 Express（Node.js）和 FastAPI（Python）分别提供不同业务能力。两者通过 REST API 进行增删改查操作，通过 SSE 协议实现 AI 对话的流式输出。

### 2.1.2 项目组成

| 模块 | 技术栈 | 职责 |
|------|--------|------|
| LifePilot | Next.js 15, React 19, MobX | 前端界面 |
| LifePilotServer | Express 5, LangGraph | AI 对话核心 |
| LifePilot_mcp | Express, Prisma | MCP 协议服务，操作 MySQL |
| ai-server | Python, FastAPI, Qdrant, Neo4j | RAG 知识库、ASR/TTS |
| Metaphorical | Next.js 16, MongoDB | 日志看板 |
| OmniBase | Next.js 16, mysql2/redis/mongodb | 多数据源管理 |
| yanmengs-ragPackage | TypeScript | 前端 SDK |
| yanmengs-logs | TypeScript | 日志 SDK |

各模块独立部署，通过 REST API、SSE 或 MCP 协议通信。开发环境通过 localhost 访问，生产环境通过 Nginx 域名转发。

### 2.1.3 Next.js 前端框架

Next.js 是基于 React 的全栈框架，提供了服务端渲染（SSR）、静态站点生成（SSG）、API Routes 等功能。

路由系统方面，Next.js 采用文件路由约定，根据 `app` 目录下的文件结构自动生成路由。

服务端组件与客户端组件划分是重要概念。React 组件默认在服务端渲染，适合展示静态内容和获取数据库数据。对于需要浏览器 API（如 WebRTC、localStorage）或处理用户交互的组件，需要显式添加 `"use client"` 指令标记为客户端组件。

### 2.1.4 Express 后端服务

Express 是 Node.js 生态中最流行的 Web 应用框架。

REST API 接口用于处理任务管理、用户认证等常规业务。Express 的路由系统和中间件机制使得 API 开发清晰有序。

SSE（Server-Sent Events）用于实现 AI 对话的流式输出。服务端通过 `Transfer-Encoding: chunked` 和 `Content-Type: text/event-stream` 头部将响应变为流式模式，持续向客户端发送数据。

### 2.1.5 状态管理

前端状态管理采用 MobX 6。MobX 提供了响应式编程模型，通过装饰器或原生 API 将 JavaScript 对象转换为可观察对象。当可观察属性变化时，依赖这些属性的 UI 组件会自动更新。

### 2.1.6 SSE 通信机制

SSE 是一种服务端向浏览器推送数据的技术。相比 WebSocket，SSE 基于 HTTP 协议，通信是单向的（服务端到客户端），实现更简单；支持自动重连，适合长时间连接。

本项目使用 SSE 实现 AI 回复的流式输出。前端根据 SSE 事件的 type 字段区分不同内容：type 为 think 表示 AI 正在推理，type 为 response 表示回复文本，type 为 confirm 表示需要用户确认，type 为 done 表示对话结束。

## 2.2 智能Agent架构与多Agent协同机制

### 2.2.1 大语言模型与提示词工程

大语言模型（LLM）是 Agent 系统的核心能力来源。本项目使用 GPT-4o 作为主力模型处理复杂推理任务，使用 DeepSeek 进行轻量级的意图识别。

提示词工程是调用 LLM 的关键技术，通过设计合适的指令模板、上下文信息和输出格式引导模型产生预期结果。

### 2.2.2 LangGraph 工作流框架

LangGraph 是 LangChain 团队开发的工作流框架，用于构建有状态、多步骤的 Agent 应用。与 LangChain 的 LCEL 链式调用不同，LangGraph 采用图结构来组织 Agent 逻辑，支持条件分支、循环、人为中断等复杂控制流。

State（状态）是贯穿整个工作流的共享数据结构。工作流每处理一个节点后会返回新的 State，用于下一步处理。

Node（节点）是工作流中的处理单元。每个节点是一个函数，接收当前 State，返回需要更新的 State 字段。

Edge（边）连接两个节点，定义了工作流的执行顺序。普通边表示无条件跳转，条件边根据 State 内容判断下一步走向哪个节点。

Checkpoint 用于保存和恢复工作流状态。本项目在用户确认环节使用 Checkpoint 保存当前进度，用户确认或拒绝后从检查点恢复继续执行。

### 2.2.3 MCP 协议

Model Context Protocol（MCP）是一种标准化协议，用于让 AI 模型与外部工具和服务进行交互。MCP 采用客户端-服务端架构：AI 应用内嵌入 MCP 客户端，通过 JSON-RPC 格式发送工具调用请求，服务器执行后返回结果。

本项目将任务管理功能封装为 MCP 服务（LifePilot_mcp），AI Agent 通过 MCP 协议调用 create_task、get_tasks、update_task、delete_task 等工具操作 MySQL 数据库。这种设计有两个好处：一是数据库操作权限收口到独立服务，AI 无法执行预定义之外的 SQL；二是工具接口标准化，便于扩展更多工具。

### 2.2.4 意图识别与路由

用户对话入口是 Router 节点，负责判断用户意图并分发到对应的工作流。Router 节点接收用户输入后，调用 DeepSeek 模型进行分类，输出意图标签（如 build_todo、what_to_do、travel、howToGo、rag）。

### 2.2.5 三角色协作模式

本项目在任务规划和任务操作场景中采用三角色协作模式，包括执行者（executor）、检查者（inspector）和管家（manager）。

执行者负责理解用户需求并生成结构化输出。以新建任务规划为例，执行者分析用户描述的时间、任务类型、优先级等信息，生成包含任务列表和建议的计划文本。

检查者验证执行者输出的质量。检查者 Prompt 中定义了质量标准，检查执行者的输出是否完整、合理、格式正确。检查通过则进入下一环节，不通过则返回执行者重新生成，形成循环直到通过。

管家角色以自然语言与用户沟通。他整合执行者和检查者的结论，用友好的方式呈现给用户。

### 2.2.6 ReAct 推理模式

ReAct（Reason + Act）是一种让 Agent 自主推理和工具调用的模式。与三角色模式不同，ReAct 模式下 LLM 被赋予工具调用权限，可以在推理过程中自主决定何时调用工具、调用什么工具。

ReAct 的工作循环为：LLM 生成推理步骤和工具调用请求，如果需要工具则执行后获取结果，LLM 根据结果继续推理或生成最终回复。

本项目在出行规划和路线查询场景中使用 ReAct 模式，接入了高德地图 MCP 和小红书 MCP。Agent 可以自主调用地图工具查询地点、计算路线，也可以调用小红书工具搜索旅游攻略。

### 2.2.7 人在回路机制

人在回路（Human-in-the-Loop）是指在自动化流程中插入人工决策点。本项目在任务规划场景实现了人在回路：AI 生成计划后暂停，等待用户确认。

LangGraph 的 interrupt() 函数用于中断工作流。调用 interrupt() 后，工作流暂停并保存检查点。前端收到中断事件后展示确认界面，用户操作后前端调用 resume 接口并传递用户决策，后端从检查点恢复工作流继续执行。

## 2.3 WebRTC与音视频记录相关技术

### 2.3.1 浏览器音视频采集

浏览器通过 MediaDevices API 访问用户设备的摄像头和麦克风。getUserMedia() 方法请求用户授权后，返回包含音视频轨道的 MediaStream 对象。

本项目使用 MediaRecorder API 将 MediaStream 录制为音频文件。MediaRecorder 支持多种编码格式，本项目使用 webm 格式。录制完成后，生成的 Blob 对象可以直接上传到服务器或进行后续处理。

### 2.3.2 语音识别技术

语音识别（ASR）将语音信号转换为文本。本项目的 ASR 流程为：前端采集音频并上传到 ai-server，ai-server 调用 SiliconFlow 的 SenseVoiceSmall 模型进行转写。

SenseVoiceSmall 是硅基流动推出的轻量级语音识别模型，支持中文普通话的实时转写。

前端通过 MediaRecorder 采集音频后，以 multipart/form-data 格式 POST 到 ai-server 的 /async/transcribe 接口。服务器返回识别结果后，前端将文本填入对话框并触发 AI 对话流程。

### 2.3.3 语音合成技术

语音合成（TTS）将文本转换为语音输出。本项目采用后端代理模式：前端将需要播报的文本发送给 Express 后端，后端调用 ai-server 的 TTS 接口获取音频流，再转发给前端播放。

后端代理模式有两个考虑：一是 ai-server 的 API Key 不需要暴露给浏览器；二是可以在后端对文本进行预处理，过滤掉 Markdown 格式符号、代码块等不适合播报的内容。

### 2.3.4 流式音频传输

流式音频传输使得 AI 回复可以边生成边播报，提升用户体验。本项目的 TTS 流式流程为：前端发送文本到后端，后端以流式方式调用 ai-server 接口，收到音频 chunk 后立即转发给前端，前端通过 Audio 标签的 src 属性指向后端接口实现边收边播。

### 2.3.5 自我记录与 Canvas 合成

自我记录功能实现了基于浏览器的视频日记录制能力，核心采用 Canvas 画布合成方案。

摄像头采集通过 MediaDevices.getUserMedia() 获取用户设备的视频轨道，获取到的 MediaStream 直接绑定到 video 元素用于预览。Canvas 绑定相同的 video 元素作为绘制源，通过 requestAnimationFrame 驱动绘制循环，将视频帧与用户添加的贴纸同步渲染到画布上。

贴纸系统采用绝对定位叠加层设计。每张贴纸记录在画布中的归一化坐标（0-1 范围）和缩放比例，绘制时根据当前视频分辨率动态转换为实际像素位置和尺寸。贴纸素材支持 PNG 透明背景，与视频帧混合后形成最终画面。

视频编码使用 MediaRecorder API 监听 Canvas 的 captureStream() 获取合成后的 MediaStream。录制过程中持续调用 canvas.captureStream(60) 保证帧率，MediaRecorder 将其实时编码为 webm 格式。音频轨道从原始摄像头 MediaStream 中提取后混入合成流，确保录制成品包含环境声音。

录制完成的 Blob 对象通过七牛云 OSS 前端直传模式上传，绕过后端服务器中转。视频元数据（OSS 地址、时长、创建时间戳）通过后端 API 持久化到 MySQL 数据库。

WebRTC 技术在本项目中预留用于后续扩展实时音视频通话场景。

## 2.4 数据存储与消息提醒相关技术

### 2.4.1 关系型数据库 MySQL

MySQL 是本项目的主数据库，存储用户账户、任务数据、标签等核心业务数据。MySQL 使用 Prisma ORM 进行访问，Prisma 提供类型安全的数据库操作接口。

本项目的数据表包括 User（用户表）、Task（任务表）、Tag（标签表）、ChatHistory（对话历史表）等。Task 表的主要字段包括 title（标题）、content（内容）、startAt（开始时间）、endAt（截止时间）、status（状态）、tipsFrequency（提醒频率）等。

### 2.4.2 文档数据库 MongoDB

MongoDB 是 NoSQL 文档数据库，本项目用它存储对话历史和系统日志。MongoDB 的灵活 Schema 适合存储结构不固定的数据，对话消息包含用户输入、AI 回复、时间戳、元数据等字段。

Metaphorical 通过 Mongoose ODM 管理日志数据，按 projectKey 隔离不同项目的日志。

### 2.4.3 缓存服务 Redis

Redis 是内存键值存储，本项目用它实现会话缓存和任务调度队列。

在会话缓存场景，Redis 存储用户的对话上下文和临时状态，减轻数据库压力。会话数据设置 TTL，过期后自动清理。

在任务调度场景，Redis 的 Zset（有序集合）存储待提醒任务。Zset 的 Score 为触发时间戳，系统通过 Zrangebyscore 查询当前需要提醒的任务，用 Zrem 原子性地抢占任务防止重复处理。

### 2.4.4 向量数据库 Qdrant

Qdrant 是高性能向量数据库，本项目用它实现 RAG 知识库的语义搜索。与早期的 Weaviate 方案相比，Qdrant 在向量检索性能和过滤能力上有明显优势。

用户上传的文档经过解析、分块、嵌入后存储到 Qdrant，检索时将用户问题也转为向量，通过向量相似度找到最相关的文档块。Qdrant 支持混合检索（结合关键词和向量），支持按用户 ID 隔离数据保证隐私，支持添加自定义元数据用于过滤。

### 2.4.5 图数据库 Neo4j

Neo4j 是本项目引入的图数据库，用于存储知识图谱。文档入库时，LLM 会从文本中抽取实体和关系三元组，存入 Neo4j 图数据库。

图检索是 RAG 多路召回的重要组成部分。当用户提问时，系统不仅做向量检索，还会从 Neo4j 中查询相关的实体关系，补充到上下文中。

### 2.4.6 对象存储服务

七牛云 OSS 提供文件存储和 CDN 加速服务。本项目用它存储用户上传的文档、图片等文件，以及 TTS 生成的音频文件。

文件上传采用前端直传模式：前端请求后端获取上传凭证，直接将文件 POST 到七牛云。这种模式减少了后端带宽压力，文件走 CDN 加速访问更快。

### 2.4.7 异步任务调度

任务到期提醒通过定时轮询实现。调度器每分钟运行一次，从 Redis Zset 中获取已到期的任务，抢占后查询任务详情，判断是发送提醒邮件还是标记为超时。提醒频率由用户配置，支持设置多次提醒。

邮件发送使用 Nodemailer 库，通过 QQ 邮箱 SMTP 服务发送。邮件内容使用 HTML 模板，包含任务名称、到期时间、当前状态等信息。

### 2.4.8 Prisma ORM

LifePilot_mcp 使用 Prisma 作为 MySQL 的 ORM 层。Prisma 提供类型安全的数据库操作接口，通过 schema.prisma 定义数据模型，自动生成客户端代码。

## 2.5 RAG 与知识库服务

### 2.5.1 FastAPI 框架

ai-server 是项目的 Python RAG 服务模块，基于 FastAPI 构建。FastAPI 是现代高性能 Python Web 框架，利用 Python 类型注解实现自动 API 文档生成和请求验证。

ai-server 提供以下核心接口：

| 路由 | 方法 | 功能 |
|------|------|------|
| `/documents/upload` | POST | 文件上传解析入库 |
| `/documents/process-url` | POST | URL 文件解析入库 |
| `/documents/delete` | POST | 删除知识库文档 |
| `/chat/ask` | POST | RAG 智能问答 |
| `/tts/stream` | POST | TTS 流式合成 |
| `/asr/transcribe` | POST | ASR 音频转文字 |

### 2.5.2 多路召回 RAG 架构

ai-server 实现了多路召回 RAG 架构，融合三种检索方式：

Dense 检索（向量检索）：将文本转为高维向量，在 Qdrant 中做相似度搜索。

Sparse 检索（BM25）：基于关键词的传统信息检索方法，与向量检索形成互补。

Graph 检索（图谱检索）：从 Neo4j 中查询与问题相关的实体关系路径。

三种检索结果经过 Cross-Encoder 重排后，送入 LLM 生成答案。

### 2.5.3 知识图谱抽取

文档入库时，LLM 自动从文本中抽取实体和关系三元组。Neo4j 存储抽取结果，图检索为问答提供关联信息补充。

### 2.5.4 项目级数据隔离

每个应用（app_name）使用独立的 Qdrant collection 和 Neo4j 数据，确保项目间数据完全隔离。用户在调用接口时通过 Header 传递 app_key，ai-server 根据 app_key 决定操作哪个 collection。

### 2.5.5 文档解析与多格式支持

ai-server 支持多种文档格式的解析：

| 文件类型 | 解析器 |
|---------|--------|
| PDF | PyMuPDF / pdfplumber |
| Word (.doc/.docx) | python-docx |
| Excel (.xls/.xlsx) | openpyxl |
| PPT (.pptx) | python-pptx |
| 图片 (jpg/png/bmp) | OCR（图像解析） |
| TXT / Markdown | 直接读取 |

### 2.5.6 SiliconFlow 服务集成

SiliconFlow 是本项目使用的 AI 服务平台，提供多种模型 API：

Embedding 模型：使用 SiliconFlow Qwen3-Embedding-8B 做文本向量化。

Cross-Encoder：使用 SiliconFlow cross-encoder 对检索结果进行重排。

LLM 生成：答案生成使用 MiniMax-M2.7 模型。
