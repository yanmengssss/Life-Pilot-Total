# 第2章 相关技术与开发工具

本章围绕系统实现所依赖的关键技术展开说明。LifePilot 并不是单一的任务清单应用，它同时涉及前后端协同、智能工作流编排、知识检索增强、语音处理以及多源数据存储。若只从某一个子模块解释系统实现，很难看清各部分之间的配合关系。因此，本章先介绍前后端分离架构与通信方式，再讨论智能体协作机制、音视频处理手段、数据存储方案和知识库服务，为后续需求分析与系统设计提供技术背景。

## 2.1 Web应用架构与前后端分离模式

系统面向的是持续交互型业务场景，既要支持常规的数据读写，又要处理流式对话、文件上传和异步任务。仅靠单体式页面渲染很难兼顾开发效率与服务边界，因此项目采用前后端分离架构，将界面交互、业务编排和知识服务放在不同层次处理。本节从整体架构、模块组成、前端框架、后端服务、状态管理和流式通信六个方面说明相关技术。

### 2.1.1 前后端分离架构

传统 Web 应用通常将页面渲染、业务处理和数据访问集中在同一服务中。该方案在功能较少时实现直接，但当系统需要同时处理实时交互、异构数据源和多类智能服务时，模块之间容易出现较强耦合，后续扩展也会受到限制。

前后端分离架构将界面层与服务层拆分为独立部分。前端负责页面渲染、用户交互和状态展示，后端负责业务规则、数据访问和服务编排，双方通过 HTTP 接口交换数据。对于本项目而言，这种划分不仅有利于界面和业务逻辑分别演进，也便于将任务管理、智能对话和知识库服务部署到不同运行环境中。

项目在实现上采用了典型的多服务协同方式。前端主应用基于 Next.js 构建，负责任务界面、知识库页面和对话抽屉；业务后端采用 Express 编排智能工作流与提醒流程；知识服务则由 FastAPI 提供文档处理、检索问答和语音接口。普通业务请求主要依赖 REST 方式完成，AI 对话结果则通过 SSE 逐步返回给前端。

### 2.1.2 项目组成

从仓库结构看，系统由主业务模块和若干配套模块共同构成。论文主体主要关注任务管理、智能对话、知识库与语音服务，但这些能力的落地仍依赖工具层、日志层和运维支撑。

| 模块 | 技术栈 | 职责 |
|------|--------|------|
| LifePilot | Next.js 16、React 19、MobX | 用户侧前端界面与交互 |
| LifePilotServer | Express 5、LangGraph | 智能对话编排与业务服务 |
| LifePilot_mcp | Express、Prisma、MCP SDK | 工具调用服务与任务数据操作 |
| ai-server | FastAPI、Qdrant、Neo4j | 文档处理、RAG 问答、语音服务 |
| Metaphorical | Next.js 16、MongoDB | 日志展示与运行观测 |
| OmniBase | Next.js 16、MySQL、Redis、MongoDB | 多数据源管理与调试 |
| yanmengs-ragPackage | TypeScript | 前端统一调用 SDK |
| yanmengs-logs | TypeScript | 日志采集与上报 SDK |

这些模块可以独立部署，通过 REST、SSE 和 MCP 等方式通信。开发环境下采用本地端口协作，生产环境中则通过反向代理完成统一入口转发。

### 2.1.3 Next.js 前端框架

Next.js 是建立在 React 之上的全栈框架，兼顾页面渲染、路由组织和服务端能力封装。对于本项目而言，选择 Next.js 的原因不只在于页面开发效率，还在于它允许前端界面、局部服务逻辑和构建流程保持在同一工程内，便于管理登录、任务、知识库和对话等多类页面。

在路由组织方面，Next.js 基于目录结构自动生成页面路径，能够减少传统前端项目中额外的路由配置工作。项目采用 App Router 组织页面，将任务列表、日历、知识库、自我记录和数据分析等功能拆分为相对独立的页面模块，这种结构与系统业务边界较为一致。

Next.js 还引入了服务端组件与客户端组件的划分方式。静态展示、数据预取和不依赖浏览器能力的部分适合放在服务端完成，而录音、视频采集、拖拽交互和本地存储等功能则必须由客户端组件处理。对于本项目中的语音采集、对话抽屉和自我记录模块，这种划分能够减少首屏负担，同时保留前端交互的灵活性。

### 2.1.4 Express 后端服务

Express 是 Node.js 生态中使用广泛的 Web 框架，适合构建中间层服务和业务编排接口。项目中的业务后端并不承担复杂页面渲染任务，而是主要负责对话入口、工作流调度、提醒处理、工具调用和结果转发，因此 Express 的轻量特点与该场景较为匹配。

在常规业务处理中，Express 通过路由与中间件机制组织用户认证、任务操作和会话管理等逻辑，结构较清晰，也便于后续插入日志、鉴权和异常处理。对于需要跨服务转发的场景，例如语音播报或知识问答请求，Express 还承担了统一入口的作用，减少了前端直接访问多个后端所带来的复杂度。

除普通请求外，Express 在本项目中还承担了流式响应任务。AI 对话结果并不是一次性返回的完整文本，而是随着模型推理逐步输出到前端。这要求服务端能够维持长连接并持续推送内容，SSE 正是在该环节被引入，用于支撑对话界面的实时反馈。

### 2.1.5 状态管理

前端状态管理采用 MobX。相较于更强调模板化约束的方案，MobX 更接近响应式对象模型，适合处理任务列表、标签筛选、用户信息和对话面板这类变化频繁且彼此关联的界面状态。

在 MobX 模式下，状态以可观察对象形式维护，页面组件根据依赖关系自动更新。对于任务增删改查、筛选条件切换、知识库文件状态刷新和对话面板展开收起等场景，这种方式能够减少大量手工同步代码，也让视图层与业务状态保持较稳定的对应关系。

本项目之所以采用 MobX，还与界面结构有关。系统既包含常规表单和列表，也包含带流式更新特征的对话界面。若状态管理过于分散，前端很容易出现刷新不一致的问题；而集中维护全局状态后，不同页面和组件之间的数据传递路径会更清楚。

### 2.1.6 SSE 通信机制

SSE 是一种基于 HTTP 的服务端单向推送机制，适合处理服务器持续产生文本结果、客户端按顺序接收的场景。与双向通信更强的 WebSocket 相比，SSE 的实现门槛更低，连接管理也更直接，对于 AI 对话这种“服务端持续输出、客户端逐步展示”的业务尤其合适。

本项目将 SSE 用于智能对话结果传输。后端在模型生成过程中不断推送阶段性内容，前端根据事件类型区分思考过程、正式答复、人工确认信号和结束标记，并据此更新界面。这样处理的好处在于，用户不必等待完整文本生成后再看到结果，系统的交互节奏也更接近真实会话。

从工程实现看，SSE 还承担了工作流状态传递的作用。任务规划场景中，系统需要在中间阶段插入用户确认节点，若采用普通同步请求，则很难自然表达“先生成，再确认，再继续执行”的处理流程。SSE 则为此类中断式交互提供了较合适的通信基础。

## 2.2 智能Agent架构与多Agent协同机制

本系统的智能能力并非简单建立在单轮问答上，而是由意图判断、任务规划、工具调用和人工确认共同组成。直接依赖单模型生成虽然实现简单，但在任务创建、出行规划和知识问答等场景中，输出稳定性和可控性都不足。本节介绍支撑智能层实现的关键方法，包括大语言模型、LangGraph 工作流、MCP 协议、意图路由、三角色协作、ReAct 模式以及人在回路机制。

### 2.2.1 大语言模型与提示词工程

大语言模型是系统智能能力的基础。它不仅承担自然语言理解任务，还负责在上下文条件下生成任务建议、知识问答结果和结构化中间输出。项目中将不同模型分配给不同任务：复杂生成与推理场景由能力更强的模型处理，轻量分类与路由则交给成本较低的模型完成。

提示词工程在这里起到约束模型行为的作用。由于模型天然倾向于生成开放文本，如果缺乏明确约束，就容易出现格式不稳定、内容冗余或偏离任务目标的问题。因此，系统通过角色设定、输入上下文、输出结构要求和质量约束来引导模型，让其在任务规划、检查和答复生成时分别承担不同职责。

在论文所讨论的场景中，提示词并不是简单的问答模板，而是工作流节点的行为边界。模型在不同节点接收到的上下文并不相同，输出目标也不同，这种按节点组织提示信息的方式为后续工作流拆分提供了基础。

### 2.2.2 LangGraph 工作流框架

LangGraph 是面向有状态智能工作流的编排框架，适合组织多步骤、可分支、可恢复的处理流程。相比顺序执行的链式结构，它更强调节点之间的状态传递与条件跳转，因此能够较自然地表示意图路由、循环检查和人工确认等过程。

在 LangGraph 中，状态贯穿工作流始终，节点负责读取当前状态并返回更新结果，边则决定节点之间的执行关系。这样的结构更接近“先判断，再分流，再回收结果”的业务流程，而不是单线条生成。对本项目而言，任务规划、路线查询和知识问答都需要根据上下文不断调整后续步骤，因此图结构比简单调用链更合适。

项目还利用了 LangGraph 的状态保存与恢复能力。任务规划进入用户确认阶段时，系统会先保留当前工作流进度，待用户作出选择后再继续后续处理。若没有这类机制，任务生成结果只能一次性提交，难以在“自动生成”和“人工决定”之间建立清晰边界。

### 2.2.3 MCP 协议

Model Context Protocol 是一种面向模型工具调用的标准化协议，用于让智能体以受控方式访问外部能力。其基本思路是把可调用工具封装为独立服务，由客户端统一发起请求并接收结果，而不是让模型直接接触数据库或任意执行环境。

本项目将任务创建、任务查询、任务更新和标签操作封装在独立工具服务中，业务后端通过 MCP 客户端调用这些能力。这样处理有两个直接作用：一方面，数据写入权限被收拢到工具层，智能工作流无法越过预设接口直接操作数据库；另一方面，工具边界被标准化后，后续接入更多业务能力也更方便。

从系统设计角度看，MCP 将“模型负责决策”与“服务负责执行”分开了。模型输出的不是最终数据库操作，而是一个受限工具请求，这种机制有助于降低错误写入和越权访问的风险。

### 2.2.4 意图识别与路由

用户输入进入系统后，并不会直接交给某一个固定工作流处理，而是先经过意图识别环节。原因在于，同样是自然语言输入，背后可能对应任务创建、行动建议、出行规划、路线查询或知识问答，不同任务需要的上下文和执行逻辑并不一致。

项目在对话入口设置了路由节点，用较轻量的模型先判断用户意图，再把请求分发到合适的处理链路。这一步的价值不在于提高模型“理解力”，而在于减少主工作流的无效调用。若所有请求都交由同一条链路处理，系统不仅成本更高，答复风格和输出结构也会变得不稳定。

从实现效果看，意图路由相当于智能层的第一道分流机制。它决定后续是进入任务规划流程、行动建议流程，还是转入外部工具参与更强的出行或知识处理链路。

### 2.2.5 三角色协作模式

在任务规划与任务操作场景中，项目没有采用单模型一次性生成结果，而是引入执行者、检查者和管家三类角色分工协作。这样设计的主要原因是，任务生成既要考虑结构化准确性，又要兼顾可读性，仅靠一个角色同时完成所有目标，结果往往不稳定。

执行者负责理解用户需求并生成结构化结果。以任务规划为例，该角色需要从自然语言描述中提取时间、事项类型、优先级等信息，形成便于后续处理的中间表示。检查者则根据预设规则核对执行结果，判断内容是否完整、逻辑是否合理、格式是否符合要求；若发现问题，流程会回到上一阶段继续修正。

管家角色承担面向用户的表达工作。它不直接决定任务是否成立，而是将中间结果整理为更适合阅读和确认的文本。三类角色分开后，系统在准确性、可读性和交互友好度之间更容易取得平衡。

### 2.2.6 ReAct 推理模式

ReAct 将推理过程与工具调用结合在同一条链路中。模型在生成答案前，可以先判断是否需要访问外部工具；得到工具结果后，再继续后续推理。这种模式更适合需要外部信息支持的问题，例如地点查询、路线规划和旅游攻略整理。

与三角色协作模式相比，ReAct 更强调“边推理、边行动”。前者适合结构化任务规划和质量审查，后者则更适合开放式问题处理，尤其是在模型必须借助地图服务或外部内容源才能给出答案的情况下。

本项目在出行规划相关场景中采用了该模式。系统接入地图工具和内容检索工具后，模型可以根据问题需要自主选择工具，再结合返回结果组织最终答复。这样不仅减少了纯语言猜测，也让路线与地点建议更贴近实际可用信息。

### 2.2.7 人在回路机制

人在回路机制的目的，是在自动化处理流程中保留人工决定权。对于任务规划场景而言，模型可以提供建议，但是否真正写入用户任务列表，仍应由用户作出最终判断。

项目在任务生成链路中设置了人工确认节点。系统先输出规划结果，再暂停当前工作流，前端据此展示确认界面；用户作出接受或拒绝的决定后，后端再恢复流程并继续后续步骤。这样一来，模型的作用被限定在“生成建议”和“准备执行”阶段，而不是直接替代用户完成业务决策。

该机制对于系统可控性很关键。若缺少人工确认，任务列表中的新增和修改将过度依赖模型判断，既不利于用户信任建立，也容易放大生成误差对业务数据的影响。

## 2.3 WebRTC与音视频记录相关技术

系统除文本交互外，还提供录音转写、语音播报和视频记录等能力。这部分实现依赖浏览器端的媒体采集机制，也依赖后端语音服务和对象存储支持。本节说明浏览器音视频采集、语音识别、语音合成、流式音频传输以及自我记录模块中使用的画布合成方案。

### 2.3.1 浏览器音视频采集

现代浏览器提供了标准化的媒体设备访问能力，用户授权后即可从摄像头和麦克风中获取实时音视频流。该机制为语音输入、自我记录和后续视频处理提供了统一入口，也避免了额外安装本地采集程序的复杂度。

本项目在录音与录屏功能中都依赖浏览器原生媒体采集接口。采集到的媒体流一部分用于页面实时预览，另一部分交由浏览器录制能力处理，最终形成可以上传的音频或视频对象。由于整个过程在浏览器端完成，交互路径较短，用户体验也更连贯。

与传统桌面采集方案相比，浏览器侧采集更适合本项目这种以 Web 为主要载体的应用。系统只需在需要的页面激活媒体权限，即可完成录音和视频记录功能。

### 2.3.2 语音识别技术

语音识别的目标，是将用户口述内容转为可继续参与对话的文本。对本项目而言，语音输入并不是独立功能，而是任务对话与知识问答的另一种输入方式，因此转写结果必须能够自然接入后续智能流程。

系统的处理链路是先由前端采集音频，再将音频上传至语音服务模块，由后端调用识别模型完成转写。项目选用的是适合中文场景的轻量级识别模型，用于处理普通话输入。识别完成后，文本结果会回填到对话输入区域，再进入与文本输入相同的后续链路。

这样设计的好处在于，语音输入与文本输入共用同一套对话逻辑，系统不需要为两类输入方式分别维护两套业务流程。

### 2.3.3 语音合成技术

语音合成用于把系统生成的文本答复转为可播放音频，从而形成更完整的多模态交互体验。对于以对话为核心的个人助理系统来说，语音播报能够减少用户持续盯屏阅读的负担，尤其适合移动场景和任务提醒场景。

本项目采用后端代理方式组织语音合成流程。前端把待播报文本发送到业务后端，由业务后端再请求语音服务接口获取音频流，随后将结果转发给浏览器播放。这样做可以避免把第三方服务密钥暴露在前端，也便于后端在转发前对文本进行预处理，去掉不适合直接朗读的标记内容。

语音合成在本系统中并不单独工作，而是嵌入对话链路。模型输出文本后，前端既可以展示文本，也可以按需触发播报，这让文本和音频两种输出方式能够并行存在。

### 2.3.4 流式音频传输

若语音播报必须等待完整音频生成后再播放，用户会明显感受到响应延迟。流式音频传输的作用就在于，将后端持续生成的音频片段尽早交付给前端，使播放过程能够更早开始。

项目中的语音播报采用分段转发方式。前端发起播报请求后，业务后端持续接收语音服务返回的数据片段，并同步转交给浏览器。前端再利用浏览器音频播放能力边接收、边播放，从而缩短从文本生成到实际播报之间的等待时间。

对于对话系统来说，这种处理方式比一次性返回整段音频更贴合连续交互场景，也更容易与文本流式输出保持节奏一致。

### 2.3.5 自我记录与 Canvas 合成

自我记录模块需要同时处理视频预览、贴纸叠加和录制输出，因此项目没有直接录制原始摄像头画面，而是采用画布合成方案。摄像头视频先作为底层画面输入，再由前端把贴纸等元素按位置绘制到画布上，最终录制的是已经完成合成的结果。

贴纸元素采用归一化坐标与缩放比例进行描述，这样做的目的是让画布内容在不同分辨率下仍能保持相对一致的位置关系。实际绘制时，系统再根据当前画面尺寸换算为像素坐标，从而完成贴纸与视频帧的同步叠加。

录制阶段，浏览器从画布中提取合成后的视频流，并与原始音频轨道合并，随后编码生成视频文件。录制结束后，文件通过对象存储直传方式上传，后端仅保存视频地址、时长和时间信息等元数据。该设计把浏览器端的媒体处理能力利用起来，减轻了后端在大文件传输中的压力。

## 2.4 数据存储与消息提醒相关技术

系统既包含任务、标签、用户等结构稳定的数据，也包含聊天消息、日志、向量索引和提醒队列等形态差异较大的数据。单一存储方式难以兼顾这些需求，因此项目采用关系型数据库、文档数据库、缓存服务、向量数据库和图数据库协同工作的方案。本节介绍各类存储组件及其在消息提醒中的作用。

### 2.4.1 关系型数据库 MySQL

MySQL 是项目的主业务数据库，主要保存用户账户、任务、标签和会话索引等结构明确、关系清晰的数据。这类数据通常需要事务支持、条件查询和稳定约束，关系型数据库更适合承担该部分工作。

项目通过 Prisma 访问 MySQL。Prisma 将数据模型定义与数据库操作接口结合起来，一方面降低了手写查询的复杂度，另一方面也让前后端业务层可以在较稳定的结构上开展开发。对毕业设计场景而言，这种方式有利于兼顾开发效率与数据层一致性。

在具体业务中，任务数据仍是 MySQL 的核心内容。任务的标题、描述、时间信息、状态和提醒设置等都需要长期保存，并与用户、标签等实体建立关联，因此采用关系型建模更便于后续管理。

### 2.4.2 文档数据库 MongoDB

MongoDB 适合处理结构变化较多、字段并不固定的数据。项目中，聊天消息明细和运行日志都具有这一特点：不同类型的消息可能附带不同元数据，日志内容也会随模块变化而扩展，因此使用文档数据库更灵活。

聊天模块中，MongoDB 主要负责保存消息内容本身，包括用户输入、模型答复以及部分附加信息；而会话级别的索引和标题则保留在关系型数据库中。这样的划分说明，系统并未简单按模块拆库，而是根据数据结构特征选择存储介质。

日志平台同样依赖 MongoDB。不同项目的日志通过项目标识区分，查询和展示则由日志看板模块统一完成，这为系统运行观测提供了基础。

### 2.4.3 缓存服务 Redis

Redis 是以内存为基础的键值存储，适合承担临时状态保存和高频访问场景。项目中，它一方面用于缓存验证码、会话状态等短期数据，另一方面用于维护任务提醒的调度队列。

在会话相关场景下，Redis 可以减少频繁访问数据库所带来的开销。对于具有时效性的状态，例如登录验证或短周期上下文信息，缓存方案比直接落库更合适，也更便于设置自动过期。

在提醒任务中，Redis 的有序集合结构尤为关键。系统按照时间顺序组织待触发任务，再由调度模块定期读取到期条目并执行后续操作，这种结构既简单，也适合提醒业务的顺序化处理需求。

### 2.4.4 向量数据库 Qdrant

Qdrant 是项目知识库服务中的主要向量存储组件，用于支撑语义检索。文档被切分并嵌入为向量后，会被写入向量数据库；用户提问时，系统再把问题转换为向量，并根据相似度找到相关文本片段。

选择 Qdrant 的原因在于，它更适合处理向量检索与过滤条件并存的场景。知识库问答不仅要找到语义相近内容，还要结合用户或应用范围进行筛选，因此向量检索能力与元数据过滤能力同样重要。

在本项目中，Qdrant 主要承担“召回相关文档片段”的工作。它并不直接生成答案，而是为后续重排与生成模块提供候选上下文。

### 2.4.5 图数据库 Neo4j

仅依赖向量检索时，系统更容易找回语义接近的文本片段，但对实体之间的关系表达仍有限。为补足这部分信息，项目引入 Neo4j 保存文档抽取得到的实体关系。

文档入库过程中，系统会从文本中识别实体及其关系，并组织为图结构。后续问答时，图数据库能够提供与问题相关的实体路径和关系补充，使检索结果不只停留在片段相似层面，也能体现知识之间的关联。

因此，Neo4j 在项目中的作用并不是替代向量库，而是作为多路召回中的另一条补充通道，与向量检索形成互补。

### 2.4.6 对象存储服务

对象存储用于保存系统中的大文件资源，包括知识库原始文档、用户上传图片、录制视频以及部分语音文件。这类资源体积较大、访问方式以文件地址为主，不适合直接存入业务数据库。

项目采用前端直传对象存储的方式组织上传流程。前端先向后端申请上传凭证，再直接把文件发送到对象存储平台。这样可以减轻业务后端在大文件中转时的带宽压力，也便于后续通过 CDN 提高访问效率。

文件本体进入对象存储后，业务后端只保存元数据与引用地址。该思路与数据库保存结构化信息、对象存储保存文件内容的分工保持一致。

### 2.4.7 异步任务调度

任务提醒不适合在页面请求中同步完成，因为提醒往往发生在用户离开页面之后。项目因此采用独立调度机制处理提醒和超时检查，由服务端定期轮询任务队列，再根据任务状态决定是否触发邮件通知。

调度器运行时会从 Redis 中读取当前到期的任务，再结合数据库中的任务详情进行判断。若任务尚未完成且达到提醒条件，则发送邮件；若任务已经超出时限，则执行相应状态处理。提醒频率由用户配置决定，因此同一任务可能在不同时间点被再次加入调度范围。

这种异步处理方式将用户交互与后台提醒解耦，避免把时间驱动逻辑压到前端，也让提醒能力可以在服务端持续工作。

### 2.4.8 Prisma ORM

Prisma 是项目关系型数据访问层的重要工具。它通过模式定义描述数据库结构，再自动生成可调用的客户端接口，从而减少手写数据库访问代码带来的重复工作。

对本项目而言，Prisma 的价值主要体现在两点。其一，数据模型和业务代码之间保持较明确映射，便于后续维护与调整；其二，工具服务层和业务后端都可以在统一数据模型上工作，避免不同模块各自维护一套访问逻辑。

从开发过程看，Prisma 更适合作为系统的数据访问中介，而不是单纯的查询封装工具。它让业务层能够把更多精力放在任务流程和服务协同上。

## 2.5 RAG 与知识库服务

知识库服务承担的是“从原始文档到可检索知识”的整条处理链路。系统不仅要支持多格式文档上传，还要完成解析、切分、嵌入、检索、重排和答案生成，因此需要较清晰的服务划分。本节介绍知识库服务所依赖的 FastAPI 框架、多路召回 RAG 架构、知识图谱抽取、项目级隔离、文档解析和 SiliconFlow 模型服务。

### 2.5.1 FastAPI 框架

ai-server 是项目中的 Python 知识服务模块，基于 FastAPI 构建。FastAPI 适合组织文档处理、检索问答和语音接口这类以数据交互为主的服务，其类型注解机制也有助于减少参数校验和接口文档维护的额外成本。

在系统中，ai-server 主要承担三类任务：一是接收文档并完成解析入库；二是根据用户问题执行知识检索与问答；三是提供语音识别和语音合成相关能力。业务后端与前端并不直接处理这些计算密集型任务，而是通过统一接口调用知识服务模块。

对应的核心接口可以概括为以下几类。

| 接口类别 | 作用 |
|------|------|
| 文档上传与处理接口 | 接收本地文件或远程文件地址并完成入库 |
| 文档删除接口 | 删除指定知识库中的文档数据 |
| 问答接口 | 基于知识库执行检索增强问答 |
| 语音合成接口 | 将文本转为可播放音频流 |
| 语音识别接口 | 将上传音频转换为文本 |

### 2.5.2 多路召回 RAG 架构

RAG 的核心不只是“先检索，再生成”，关键在于检索结果是否足够贴近问题。若只依赖单一召回方式，系统往往会在召回覆盖率和结果精度之间出现偏差。项目因此采用多路召回结构，将向量检索、关键词检索和图谱检索结合起来。

向量检索负责捕捉语义相近的文本片段，适合处理表达方式不同但含义接近的问题；关键词检索保留了传统信息检索对原词匹配的敏感性，能够弥补语义检索对部分关键术语的不稳定；图谱检索则从实体关系角度补充上下文，适合处理关联较强的问题。

三路召回得到的候选结果不会直接送入生成模型，而是先经过重排步骤重新评估相关性，再组织为回答所需的上下文。这样处理后，最终答案更依赖经过筛选的检索证据，而不是单纯依赖模型内部记忆。

### 2.5.3 知识图谱抽取

知识图谱抽取发生在文档入库阶段。系统在解析文本后，会进一步识别其中的实体和关系，并将抽取结果写入图数据库。这样一来，知识库不只保存原始片段，也保存片段之间的结构性联系。

对于问答任务而言，图谱抽取的意义在于补足“谁与谁有关、关系是什么”这类关系信息。当用户问题涉及人物、地点、事件或概念之间的联系时，图谱检索能够提供比普通文本召回更清晰的关系线索。

因此，知识图谱抽取不是独立展示模块，而是服务于后续问答质量的底层过程。

### 2.5.4 项目级数据隔离

知识库服务需要面对多应用或多项目并存的场景。若所有数据写入同一检索空间，后续问答容易出现跨项目召回，进而影响结果可靠性。项目因此采用应用级隔离策略，对不同业务来源分别维护检索空间。

系统通过应用密钥识别请求来源，再把对应文档写入各自的向量集合和图数据范围中。这样做的直接作用，是让不同项目在共享同一套知识服务时仍保持数据边界清晰。

对于面向个人或多租户的知识库系统而言，这种隔离机制是必要的基础设计，而不是附加功能。

### 2.5.5 文档解析与多格式支持

知识库系统若只支持单一文件格式，实际可用范围会受到很大限制。项目因此在文档处理层支持多种常见文件类型，使用户可以直接导入日常资料，而不必先手动转换格式。

当前支持的主要文档类型如下。

| 文件类型 | 解析方式 |
|---------|--------|
| PDF | PDF 文本抽取工具 |
| Word 文档 | Office 文档解析工具 |
| Excel 表格 | 表格数据读取工具 |
| PPT 文档 | 演示文稿解析工具 |
| 图片文件 | OCR 图像文字识别 |
| TXT / Markdown | 纯文本直接读取 |

多格式支持的意义不只在于“能导入更多文件”，更在于让知识库可以接纳真实使用场景中的原始资料。后续切分、嵌入和检索流程，都是在该前提上展开的。

### 2.5.6 SiliconFlow 服务集成

项目中的嵌入生成、结果重排和部分模型调用依赖 SiliconFlow 提供的模型服务。这样做的原因在于，知识库问答链路涉及多类模型能力，若全部在本地维护，不仅部署复杂，也会增加资源负担。

在当前实现中，SiliconFlow 主要承担三类工作。其一是文本向量化，用于把文档片段和问题映射为向量表示；其二是对候选检索结果进行重排，帮助系统从多路召回结果中选出更相关的内容；其三是支撑部分生成任务，为问答结果组织提供模型能力。

通过统一接入模型服务平台，项目将注意力更多放在业务链路和系统协同上，而不是陷入多模型底层部署细节。对于毕业设计规模的系统，这种选择更符合工程实现的实际需求。
